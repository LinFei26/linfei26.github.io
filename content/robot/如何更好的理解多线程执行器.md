## 一、从单线程执行器开始（ `rclpy.spin(node)`）

假设你是一个人（**单线程**），面前摆着：

- **三个对讲机**（三个订阅者），随时可能收到消息
- **一个闹钟**（定时器），每隔一段时间响一次
- **一个电话**（服务请求），随时会响

**单线程模式**：你**一次只能处理一件事**。

- 如果第一个对讲机不断传来大量消息，你必须一条一条处理。在此期间，闹钟响了你也顾不上，电话响了也只能先挂起。
- 如果某条消息的处理特别耗时（比如要进行复杂的图像识别），那么后面的所有任务（其他对讲机的消息、闹钟、电话）都会**排队等待**，导致明显的延迟。

这就是**单线程执行器**：一个线程按顺序处理所有回调。如果某个回调函数执行时间过长，它会**阻塞**其他所有回调。

## 二、多线程执行器：增加帮手

现在，你不单干了。你请来了**三个帮手**（线程池）。

- **帮手1**：专门负责对讲机A
- **帮手2**：专门负责对讲机B
- **帮手3**：专门负责闹钟和电话

**多线程模式**：多个任务可以**同时**被处理。

- 如果对讲机A的消息处理需要很长时间，帮手1被占用，但帮手2还可以处理对讲机B的消息，帮手3可以准时响应闹钟 —— **互不阻塞**。
- 缺点是：如果帮手不够用（比如10个对讲机同时涌入大量消息），新任务需要排队等待空闲帮手。

这就是**多线程执行器**：使用多个线程并发处理回调，提高节点的**吞吐量**和**响应性**。

## 三、代码逐行解释

```python
from rclpy.executors import MultiThreadedExecutor   # 导入多线程执行器

executor = MultiThreadedExecutor()                  # 创建执行器（默认线程数 = CPU核心数）
executor.add_node(node)                             # 把你的节点交给执行器管理
executor.spin()                                     # 启动执行器（开始多线程循环）
```

与单线程版本对比：

| 代码 | 解释 |
|------|------|
| `rclpy.spin(node)` | 内部自动创建**单线程**执行器，当前线程被占用。 |
| `executor = MultiThreadedExecutor(); executor.spin()` | 手动创建**多线程**执行器，内部会启动多个线程来处理回调。 |

### 关键点

- `add_node(node)` 告诉执行器：“这个节点的所有回调函数（订阅者、定时器、服务等）**都归你调度**”。
- `spin()` 启动执行器。它不会阻塞当前线程？实际上**它会阻塞**（就像单线程的 `spin()` 一样），但内部有多个线程在干活。当前线程（主线程）会等待直到执行器被关闭。

### 调整线程数量

```python
executor = MultiThreadedExecutor(num_threads=4)   # 明确指定4个线程
```

如果不指定，默认线程数等于你的**CPU核心数**（比如4核8线程的可能默认4）。

## 四、你什么时候需要多线程执行器？

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 节点只有1~2个订阅者，回调都很轻量（如打印日志） | **单线程（默认）** | 简单，无额外开销。 |
| 节点有多个订阅者，消息频率都很高（每个10ms来一条），且回调函数本身不耗时 | **多线程** | 避免一个队列积压影响其他。 |
| 某个回调函数本身就很耗时（如图像处理、路径规划） | **不要依赖多线程** | 多线程只能缓解并发，但每个耗时回调仍会占用一个线程。更好的办法：把耗时任务挪到**另一个独立节点**中。 |
| 节点同时处理订阅、服务、动作，需要保证低延迟响应 | **多线程** | 让不同类型的工作并行。 |

## 五、多线程的潜在问题（并发引起的“打架”）

多个线程同时访问（修改）同一个变量，可能导致**数据错乱**。例如：

```python
class MyNode(Node):
    def __init__(self):
        self.counter = 0
        self.create_subscription(..., self.callback_a, ...)
        self.create_subscription(..., self.callback_b, ...)
    
    def callback_a(self, msg):
        self.counter += 1      # 线程A修改
    
    def callback_b(self, msg):
        print(self.counter)    # 线程B读取
```

如果 `callback_a` 和 `callback_b` 在不同线程中并发执行，线程B可能读到**修改了一半的值**（比如原先是0，线程A刚读到0还没完成加1），造成脏读。

**解决方案**：

- 使用 **`threading.Lock`** 保护共享变量
- 或者使用ROS2提供的**回调组（Callback Group）**，将需要互斥的回调放到同一个组中，强制它们串行执行：

```python
from rclpy.callback_groups import MutuallyExclusiveCallbackGroup

group = MutuallyExclusiveCallbackGroup()
sub1 = self.create_subscription(..., callback_a, callback_group=group)
sub2 = self.create_subscription(..., callback_b, callback_group=group)
```

这样两个回调不会同时运行。

## 六、总结（一句话）

> **单线程执行器**是你一个人按顺序干活（简单，但耗时任务会阻塞其他）；**多线程执行器**是你带一个团队同时干活（提高并发，但要注意协调共享资源）。

对于初学ROS2的你，**默认的单线程执行器已经够用**。只有当你明确观察到消息积压、响应变慢，并且确认回调函数本身足够轻量时，才值得尝试多线程执行器。如果回调函数本身就耗时就该重构代码，而不是依赖多线程。
